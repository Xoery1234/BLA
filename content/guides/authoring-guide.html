1. Quick Start — Your First Page in 5 Steps

This gets you from zero to a live preview in under 30 minutes.

  1. Open the template. Go to /content/templates/ in da.live. Open r1b-coming-soon-teaser for a teaser page, or r2a-conversion-landing for a campaign landing page.
  2. Clone it. In Sidekick, click the three-dot menu and select "Copy." Paste the copy into your target folder (e.g. /content/revlon/campaigns/spring-2026/). Rename the file to your page name (use lowercase, hyphens for spaces: spring-drop-teaser).
  3. Replace placeholder content. Every placeholder starts with a ⬇ arrow. Replace each one with your real content. Delete instruction text when done. Keep the block table structure intact — don't delete or reorder the rows.
  4. Set your metadata. Open the metadata sheet in Sidekick. Set title, description, brand, and optionally campaign. Details in the Metadata section.
  5. Preview. Click "Preview" in Sidekick. Your page renders at https://main--{repo}--{org}.aem.page/{path}. Check it on mobile too (resize your browser or use DevTools). When satisfied, follow the pre-publish checklist.
Tip: Keep the wireframe open alongside. The wireframes at /accelerator-docs/wireframes/r1b/ and /accelerator-docs/wireframes/r2a/ show what the finished page looks like with Revlon tokens applied. The canonical example at /content/revlon/examples/r1b-red-revival-teaser shows a completed page with real content. Use these as visual reference while editing.

2. Block Table Conventions

Every block in da.live is a table. This is the most important thing to understand.

How blocks work

A block table has a specific structure that the system reads to render your content. The first row of the table always contains the block name. Subsequent rows contain your content. The system matches the block name to a code template and renders your content accordingly.

ConceptWhat it meansExample
Block name (row 1)Tells the system which block to render. Must match exactly — case-insensitive but spelling matters.Hero, Intrigue, Email Capture
Content rows (2+)Your actual content. Each row maps to a slot in the block. Order matters.Row 2 = image, Row 3 = text
Block variantSome blocks have variants. Add the variant name after the block name, separated by a space.Hero (cinematic)
Critical: The H1 Rule Every page must have exactly one H1 heading. It must be formatted as "Heading 1" in da.live — NOT as bold text, NOT as Heading 2. If the H1 is missing or incorrectly formatted, the page will render incorrectly and SEO will be broken. The H1 goes inside your Hero block.

What can go wrong

MistakeWhat happensHow to fix
Block name misspelledBlock renders as a plain tableCheck spelling: Intrigue not Intrige
Missing H1Page has no title, SEO failsSelect your headline text, format as Heading 1
H1 formatted as boldLooks like a heading but system ignores itRemove bold, apply Heading 1 style
Two H1s on one pageSEO confusion, accessibility violationChange the second to Heading 2
Content rows in wrong orderBlock shows content in wrong slotsFollow template row order exactly
Empty row in block tableMay cause rendering gapsDelete truly empty rows
Image not wrapped in pictureImage missing or brokenUse da.live image insert (drag or paste)
Block doesn't appear at allBlank space where block should beCheck block name spelling. Open DevTools (F12) for JS errors. Delete block, re-copy from Sidekick Library, refill.
Important: row numbering below does NOT include the block-name row. In da.live, every block table starts with a row containing the block name (e.g. "Hero", "Intrigue"). EDS strips this row before rendering. The row numbers in the block reference below refer to content rows only — what the system actually processes.

3. Block Reference

Hero

Full-width hero with background image, overlay gradient, and text content.

RowContentNotes
1Hero imageDrag image from assets panel. Min 1600×900px. Alt text required.
2Text content (all in one cell)All text in this order within a single cell: eyebrow paragraph, H1 headline, subheadline paragraph, CTA link(s). The system reads the order to assign styles. Do NOT put each text element in a separate row — they must all be in one cell.

The eyebrow is any paragraph that appears before the H1. The subheadline is the first paragraph after the H1. Links with button formatting become CTA buttons. Keep all text elements together in row 2 — splitting them into separate rows will break the hero layout.

Intrigue

Horizontal statement strip with scroll-reveal animation.

RowContentNotes
1-3Statement textOne statement per row. Max 80 chars each. 2-3 statements.

Email Capture

Single-field email signup with consent checkbox and anti-spam protection.

RowContentNotes
1Button labelWhat the submit button says. Default: "Notify me"
2Consent textLegal text for the consent checkbox. Required.
3Success messageShown after successful signup.
4Form endpoint linkWebhook URL wrapped in a link. Get this from your developer.
Form endpoint required before go-live. Without the webhook URL in row 4, email signups are lost. This is configured in config/brands/revlon/forms.json — ask your developer to set it up and give you the URL.

Social Proof

Logo strip showing press mentions or partner brands.

RowContentNotes
1Label texte.g. "As seen in" or "Featured by". Text-only row.
2+Logo imagesOne logo per row. Max 6. Use monochrome versions, 120×60px.

Columns

Multi-column layout used for Trust Row and Single Benefit blocks in R2.A.

RowContentNotes
1Column contentEach cell in the row becomes a column. Use for trust signals (3 cells) or benefit + image (2 cells).

Cards

Card grid used for customer testimonials in R2.A proof variant A.

RowContentNotes
1+Card contentEach row becomes a card. Include quote text and author attribution.

4. Image Sizing Guide

BlockDimensionsFormatMax size
Hero image1600 × 900 px (16:9)WebP or JPEG400 KB
Benefit image800 × 800 px (1:1) or 800 × 600 (4:3)WebP or JPEG300 KB
Social proof logos120 × 60 pxSVG or PNG (transparent)50 KB
OG image (social share)1200 × 630 pxJPEG300 KB

Compress images before uploading using squoosh.app (free, runs in browser). Uncompressed images slow page load and hurt your Lighthouse score.

Alt text is not optional. Every image needs descriptive alt text for screen readers. Describe what the image shows in 5-10 words. "Hero image" is not good alt text. "Revlon lipstick in bold red shade" is.

6. Campaign Palettes

Campaign palettes let you change the colour scheme of a page without touching code. Set the campaign metadata field and the palette applies on next preview.

Available palettes

Palette namePrimary colourAccent colourUse for
(empty / default)Revlon Red (#D71920)Warm Blush (#F5E6D3)Standard brand pages
holiday-2026Deep Crimson (#8B0000)Forest Green (#2D5A27)Holiday season campaigns
summer-glowCoral (#FF6B47)Golden (#FFD700)Summer campaigns

How to switch palettes

  1. Open your page in da.live.
  2. Open the metadata sheet.
  3. Find or add the campaign row.
  4. Set the value to the palette name (e.g. holiday-2026).
  5. Click "Preview" in Sidekick. The palette applies immediately.
The palette CSS file must exist. If you type a campaign name that doesn't have a matching CSS file in styles/campaigns/, the page falls back to the default brand palette. Contact your developer to create new campaign palettes.

7. Sidekick Operations

Sidekick is the browser extension toolbar that connects da.live to the delivery system. Key operations:

ActionWhat it doesWhen to use
PreviewRenders your page at .aem.pageAfter every editing session, before sharing
PublishPushes page to .aem.live (production)After pre-publish checklist is complete
EditOpens the page in da.live for editingWhen you want to change content
LibraryOpens the block libraryWhen you want to insert a new block

Using the Sidekick Library

The Library gives you drag-and-drop access to every accelerator block. No need to remember block names or table structures.

  1. Open your page in da.live.
  2. Click "Library" in the Sidekick toolbar.
  3. Browse available blocks — each shows a name and preview.
  4. Click a block to copy its table structure to clipboard.
  5. Paste into your document at the desired position.
  6. Fill in the placeholder content.

8. Preview and Publish

Preview URLs

After clicking "Preview" in Sidekick, your page is available at:

https://main--{repo}--{org}.aem.page/{your-page-path}

This URL is stable — you can bookmark it and share it with stakeholders for review. The preview updates every time you click Preview after making changes.

Live URLs

After publishing, the page is available at:

https://main--{repo}--{org}.aem.live/{your-page-path}

Or on the custom domain if configured.

Publish is live. There is no undo. Once you click Publish, the page is immediately visible to the public. There is no staging environment between preview and live. Always complete the pre-publish checklist first.

10. Pre-Publish Checklist Workflow

Before publishing any page, complete the pre-publish checklist. This is your process record that brand, legal, and technical requirements have been met.

How it works

  1. Clone the checklist template from /content/templates/pre-publish-checklist.
  2. Rename it to match your page: {page-name}-checklist (e.g. spring-drop-teaser-checklist).
  3. Work through each item. Mark each with a date and your name when complete.
  4. Share the checklist link with your approver (Sam persona).
  5. Only publish the page after all required items are checked.
This is process discipline, not a technical gate. da.live does not enforce the checklist — you can publish without completing it. But every page published without a completed checklist is a process violation. Automated approval workflow is planned for a future release.

11. FAQ — Common Mistakes

My block shows as a plain table on preview

The block name in the first row is misspelled or not recognised. Check it against the block reference above. Block names are case-insensitive but must be spelled correctly.

My hero has no headline styling

The headline text is not formatted as Heading 1. Select the text in da.live, open the paragraph styles dropdown, and apply "Heading 1." Bold text is not the same as H1.

My page looks right on desktop but broken on mobile

Check image sizes. Oversized images (4000×3000px originals) cause layout overflow on small screens. Resize to the recommended dimensions in the image guide. Also check that your text fits within the character limits — a 200-character headline wraps badly on mobile.

My campaign palette isn't applying

Three things to check: (1) the campaign metadata value matches a palette name exactly — no typos, no spaces, (2) the palette CSS file exists in styles/campaigns/, (3) you clicked Preview after saving the metadata change.

My form submissions aren't arriving

The form endpoint URL in the Email Capture block row 4 is missing or wrong. Check the URL is a valid webhook endpoint. Test by submitting your own email on preview. Check config/brands/revlon/forms.json for the correct endpoint.

I deleted a block by accident

Use da.live's undo (Ctrl/Cmd + Z) immediately. If you've already saved, you can recover from version history in da.live (File menu). As a last resort, re-copy the block from the template or Sidekick Library.
